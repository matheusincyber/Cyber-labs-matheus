# Lessons Learned

## What Worked

- Splunk Enterprise was installed successfully on Windows.
- Sysmon was installed and confirmed running.
- Sysmon Event ID 1 process creation events were visible in Event Viewer.
- Sysmon logs were ingested into Splunk.
- Windows Security logs were ingested into Splunk.
- Kali was able to scan the Windows host-only interface using Nmap.
- PowerShell-related process execution was searchable in Splunk.

## What Was Challenging

- Sysmon did not appear immediately in the Splunk local event log collection list.
- The Sysmon log source had to be added manually through Splunk configuration.
- Splunk displayed many Sysmon events as raw XML, which makes screenshots less readable.
- Network scan detection may require additional tuning or enabling Sysmon network connection logging.

## What I Would Improve Next

- Create a dedicated Splunk index named `windows` instead of using the default index.
- Enable or tune Sysmon network connection logging for stronger Nmap detection.
- Add Windows Firewall logs as an additional detection source.
- Build dashboards for PowerShell execution, failed logins, and scan activity.
- Add failed login screenshots and a complete IR-002 report.
- Add an Active Directory lab as the next project.

## Failed Login Detection

Windows Security Event ID 4625 can be used to detect failed authentication attempts. In this lab, an SMB login attempt from Kali using a nonexistent username generated a failed logon event that was ingested by Splunk. Reviewing Account Name, Source Network Address, Workstation Name, Logon Type, and Failure Reason helped confirm the activity was generated by the lab test.
